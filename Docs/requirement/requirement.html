📄 Assessment Brief

Online Banking App — Requirements Document

A clear, spoon-feeding guide to what was asked, why it was asked, and how the application is supposed to behave — written so that anyone can understand it in minutes.

What Is This Project?

This is a full-stack online banking application that lets users look up their bank account and view transactions. It is split into two parts that work together:

1
Frontend — React App The website the user sees and interacts with. It shows a form, takes inputs, calls the backend, and displays the results.
2
Backend — Spring Boot REST API The engine behind the scenes. It receives requests from the frontend, queries the MySQL database, and returns account & transaction data.
3
Database — MySQL Stores all account and transaction records. The backend reads from it using JPA/SQL queries.
💡 Think of it like a mini banking portal:
You type in your sort code + account number → the app looks up your account → shows you your balance and transaction history.

Who Asked For This & Why?

The project is an assessment task set by a company (3Di). The assessor wants to see whether the candidate can:

  • Read and understand an existing codebase written by someone else
  • Make targeted changes to both the frontend and backend
  • Connect the two parts correctly so the app still works end-to-end
  • Write unit tests to prove the new code is correct
  • Package everything with Docker so it runs with a single command
  • Comment the code so the assessor can follow the thinking
Bottom line: This is a "show us you can code and explain what you did" challenge — not just about making it work, but about demonstrating good engineering habits.

The Two Official Requirements

The brief listed exactly two things that need to be built:

Requirement 1 — Frontend Only

Show the Current Balance

The backend already sends an account.currentBalance field in its response, but the frontend was not displaying it.

What to do: Find where the account data is rendered in the React app and add a line that shows the current balance to the user.

  • No backend changes needed at all
  • Only the React component needs updating
  • The value should be readable and clearly labelled
Requirement 2 — Frontend + Backend

Filter Transactions by Date Range

Currently the app loads all transactions for an account. The requirement is to allow the user to pick a start date and an end date, and only see transactions that fall within that period.

  • Both dates are inclusive — e.g. if start = 2019-11-04, records from that exact date are included
  • If no dates are given, the existing behaviour (all transactions) still works

Requirement 2 — Broken Down Step by Step

The brief specifically says to update these layers in the backend:

A
Frontend (React) — Add Two Date Inputs The search form must have a Start Date field and an End Date field. When the user submits, these dates must be sent to the backend along with the existing sort code & account number.
B
Transaction Controller — Accept the Dates The REST endpoint that returns transactions must now accept the two new date fields in its request body.
C
Transaction Service — Filter Logic The service layer must convert the date strings into proper date-time values and pass them to the repository.
D
Transaction Repository — Database Query A new database query must retrieve only records where initiationDate is between the two given dates.
Key rule from the brief:
"The start and end dates can be inclusive of the data returned — e.g. if start date is 2019-11-04 then it should also include records which have the initiation date 2019-11-04."

What Data Does the App Work With?

The MySQL database has two tables:

Table Key Fields What it stores
account id, sort_code, account_number, current_balance One row per bank account — who owns it and how much money is in it
transaction id, source_account_id, amount, initiation_date Every deposit/withdrawal tied back to an account, with the date it happened
Test data included: The schema comes with sample accounts and transactions so you can try the app straight away without inserting data manually. Example: sort code 53-68-92, account number 73084635, balance £1071.78 with 4 transactions between April–July 2019.

How a User Should Experience the App

Here is the intended end-to-end user journey once everything is built:

1
Open the app — navigate to http://localhost:3000
2
Fill in the form — enter Sort Code, Account Number, and optionally a Start Date and End Date
3
Submit — the frontend sends a POST/PUT request to the backend
4
See results — the page now shows the Current Balance prominently, followed by a table of matching transactions
5
Optionally filter — enter dates to narrow down to a specific period; leave blank to see all transactions

Running the Application

The requirement is that the app should be easy to run. The goal is:

One command, everything starts:

docker compose up --build

This should start MySQL, the Spring Boot backend, and the React frontend — all wired together automatically. No manual database setup. No separate terminal windows. Just one command.
