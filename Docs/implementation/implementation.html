💰 Requirement 1 — Display the Current Balance

Frontend Change
🤔 What was the problem?
The backend API was already sending back an account.currentBalance value but the React component simply never rendered it on screen. The user had no way to see their balance.

📂 File Changed

⚛️ src/pages/TransactionTablePage.jsx

✏️ What we changed

After fetching the account data, the component stores the balance in state and renders it prominently above the transactions table.

// 1️⃣ New state variable to hold the balance const [currentBalance, setCurrentBalance] = useState(null); // 2️⃣ When the API responds, save the balance setCurrentBalance(response.data.account.currentBalance); // 3️⃣ Render it on the page {currentBalance !== null && ( <div className="balance-display"> 💰 Current Balance: <strong>£{currentBalance.toFixed(2)}</strong> </div> )}
✅ Result: The user can now see their account balance displayed clearly on the page as soon as they search for an account.

📅 Requirement 2 — Date Range Filter (Frontend)

Frontend Change
🤔 What was the problem?
The form only had fields for Sort Code and Account Number. There was no way for the user to say "only show me transactions from these dates."

📂 File Changed

src/pages/TransactionTablePage.jsx

What we changed

Added two state variables
startDate and endDate — both start as empty strings.
Added two date input fields to the form
Standard HTML <input type="date"> elements, bound to the state variables via onChange.
Included the dates in the API request
When submitting, startDate and endDate are added to the request body. If they are empty, the backend still returns all transactions.
// Two new state variables const [startDate, setStartDate] = useState(''); const [endDate, setEndDate] = useState(''); // Two new form inputs <input type="date" value={startDate} onChange={e => setStartDate(e.target.value)} /> <input type="date" value={endDate} onChange={e => setEndDate(e.target.value)} /> // Sent with the request body axios.put('/api/account/transactions', { sortCode, accountNumber, startDate, endDate });
✅ Result: The user picks a start date and end date, clicks Search, and only sees transactions within that window.

⚙️ Requirement 2 — Date Range Filter (Backend)

Backend Changes

The backend needed changes across four layers. Think of it like a chain — the request flows from top to bottom and we had to update each link in the chain:

1️⃣
DTO — AllTransactionsInput.java (new file)
↓
2️⃣
Controller — TransactionRestController.java
↓
3️⃣
Service — AccountService.java + TransactionService.java
↓
4️⃣
Repository — TransactionRepository.java
Layer 1 — New DTO

📄 AllTransactionsInput.java — Brand New File

A Data Transfer Object (DTO) is just a simple Java class whose only job is to carry data from the frontend into the backend. Think of it as an envelope.

We created a new DTO that extends the existing AccountInput (which already had sort code and account number), and adds two new fields:

public class AllTransactionsInput extends AccountInput { private String startDate; // e.g. "2019-04-01" private String endDate; // e.g. "2019-07-31" // Getters and Setters below... }
💡 Why extend instead of copy?
The existing AccountInput already validates the sort code and account number format. By extending it, the new DTO reuses that validation for free — no duplicate code.
Layer 2 — Controller

🎮 TransactionRestController.java

The controller is the "door" that the frontend knocks on. It reads the incoming HTTP request and decides what to do with it.

We updated the getAllTransactions method to accept our new AllTransactionsInput DTO and then branch:

// BEFORE — only accepted AccountInput public ResponseEntity getAllTransactions(@RequestBody AccountInput input) { ... } // AFTER — now accepts AllTransactionsInput (which has startDate + endDate) public ResponseEntity getAllTransactions(@RequestBody AllTransactionsInput input) { // If the user gave us dates → filter by date range if (input.getStartDate() != null && !input.getStartDate().isEmpty()) { return accountService.findTransactionsByDateRange(...); } // No dates → return everything (original behaviour) return accountService.findTransactions(...); }
✅ Backwards compatible: If the frontend sends no dates, the endpoint still works exactly as before — no breaking changes.
Layer 3 — Service

🔧 AccountService.java & TransactionService.java

The service layer contains the business logic. It knows "what should happen" — it does not talk to the database directly, it delegates that to the repository.

New method added to AccountService:

public ResponseEntity findTransactionsByDateRange( String sortCode, String accountNumber, String startDate, String endDate) { // 1. Look up the account by sort code + account number Account account = accountRepository.findBySortCodeAndAccountNumber(...); // 2. Parse the date strings into LocalDateTime objects LocalDateTime start = LocalDate.parse(startDate).atStartOfDay(); LocalDateTime end = LocalDate.parse(endDate).atTime(23, 59, 59); // 3. Ask the transaction service to get filtered transactions List<Transaction> txns = transactionService .getTransactionsByDateRange(account.getId(), start, end); // 4. Return the account + filtered transactions return ResponseEntity.ok(new AccountTransactions(account, txns)); }
💡 Why parse to LocalDateTime?
The database stores initiationDate as a timestamp. We set start to midnight (00:00:00) and end to 23:59:59 so that both dates are fully inclusive as the requirement states.
Layer 4 — Repository

🗄️ TransactionRepository.java

The repository is the only layer that talks directly to MySQL. We add a new method using Spring Data JPA's query derivation — you describe what you want in the method name and Spring writes the SQL automatically!

// Spring JPA reads this method name and generates the SQL for us: // SELECT * FROM transaction // WHERE source_account_id = ? // AND initiation_date BETWEEN ? AND ? // ORDER BY initiation_date ASC List<Transaction> findBySourceAccountIdAndInitiationDateBetweenOrderByInitiationDate( Long accountId, LocalDateTime startDate, LocalDateTime endDate );
✅ No raw SQL needed! Spring Data JPA reads the method name like a sentence and translates it into an optimised SQL query automatically.

Unit Tests — Proving It Works

6 Mockito Tests
Why unit tests?
Unit tests run without a database or server. They check that each piece of logic is correct on its own. Mockito lets us "fake" the database layer so we can test the service in isolation.

New Test File

src/test/java/.../TransactionDateRangeTest.java

The 6 Test Scenarios

# Test Name What it checks
1 getTransactionsByDateRange_returnsResults Given valid dates, the correct transactions are returned
2 getTransactionsByDateRange_emptyList If no transactions match the date range, an empty list comes back (not an error)
3 getTransactionsByDateRange_startDateInclusive A transaction on exactly the start date IS included in the results
4 getTransactionsByDateRange_endDateInclusive A transaction on exactly the end date IS included in the results
5 getTransactionsByDateRange_multipleResults Multiple transactions within the range are all returned, in order
6 findTransactionsByDateRange_accountNotFound If the account does not exist, the service returns a 404 Not Found response
✅ All 6 tests pass. They cover happy paths, edge cases, and error conditions.

Docker — One Command to Rule Them All

Infrastructure

Four new files were created to make the entire app runnable with a single command:

docker-compose.yml

The "conductor" file. Defines all three services and how they connect:

  • mysql — MySQL 8.0 database with health check
  • backend — Spring Boot (waits for MySQL to be healthy first)
  • frontend — React app served by nginx on port 3000
frontend/Dockerfile

Two-stage build:

  • Stage 1: Install Node.js, run npm install, run npm run build
  • Stage 2: Copy the built files into an nginx web server image — tiny and fast
backend/Dockerfile

Two-stage build:

  • Stage 1: Use Maven to compile the Java code into a JAR file
  • Stage 2: Copy the JAR into a slim Java runtime image and run it
nginx.conf

Tells nginx two things:

  • Requests starting with /api/ → forward to the Spring Boot backend
  • Everything else → serve the React app (with SPA routing support)

Startup Order

MySQL starts first — Docker waits until the health check passes (database is ready to accept connections)
Spring Boot starts second — only after MySQL is healthy, so no "connection refused" errors
React / nginx starts in parallel — it doesn't need the backend to start; it's just static files
App is live — open http://localhost:3000
▶️ To run the app:

docker compose up --build

To stop: docker compose down
